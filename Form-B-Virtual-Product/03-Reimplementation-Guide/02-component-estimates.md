## 分项说明

### n8n 替代引擎（20 天 / 10 天）

**什么替代的是"n8n 的工作流执行能力"，不是替代"n8n 全部功能"：**

```yaml
需要实现的核心功能:
  ✅ HTTP Webhook 监听（接收外部触发）
  ✅ 节点链式执行（Step 1 → Step 2 → Step 3）
  ✅ 条件分支（IF/ELSE 逻辑）
  ✅ HTTP Request 节点（调用外部 API）
  ✅ JSON 数据处理（提取/转换/合并）
  ✅ 执行记录存储（用于审计和重试）

不需要实现的功能（除非你有精力）:
  ❌ 可视化拖拽编辑器（n8n 的杀手功能，太难替代了）
  ❌ 400+ 预制节点（你只需要 10 个最常用的）
  ❌ 代码节点（Code Node—用外部服务代替）
  ❌ 复杂的错误重试策略（简单的 Exponetial Backoff 就够了）
```

### auth-server（5 天 / 3 天）

**这个组件在 Form-B 中已经是轻量自研（非 n8n 元素），所以重写成本相对较低：**

```yaml
工时分布:
  Day 1: RSA License Key 生成与验证（核心逻辑）
  Day 2: 7 天离线缓存机制（文件系统 + 加密）
  Day 3: API Key 两级管理（Master Key + Sub Key）
  Day 4: 管理 API 开发 + 权限校验
  Day 5: 测试 + 安全审计 + Dockerfile

核心复杂点:
  1. 机器指纹的跨平台兼容（Linux 取 /proc/cpuinfo，macOS 用 sysctl）
  2. License 的防伪设计（RSA-2048 + 过期时间 + 功能开关）
  3. 7 天离线缓存的解空间问──缓冲击后如何优雅降级
```

### ai-gateway（7 天 / 4 天）

**虽然业务目标是简单的"请求转发"，但三层安全审核的工程量不低：**

```yaml
工时分布:
  Day 1: 基础框架 + Provider 适配器（OpenAI / DeepSeek / Anthropic）
  Day 2: 鉴权层（API Key 验证 + JWT + HMAC 签名）
  Day 3: 限流层（Token Bucket + Redis 分布式限流 + 配额管理）
  Day 4: 审计层（全量日志 + 异步批量写入 + 脱敏）
  Day 5: Provider 故障切换 + 流量分发
  Day 6: 集成测试 + 安全测试
  Day 7: 性能调优 + Dockerfile + 文档

核心复杂点:
  1. 限流的原子性（Redis Lua 脚本保证并发安全）
  2. 审计日志的异步批量写入（缓冲 + 刷盘时机）
  3. Provider 健康检测（被动检测 vs 主动检测的选择）
  4. 不同 Provider 的错误映射和重试策略
```

### 前端管理面板（15 天）

**如果你已经有 n8n 的 UI 可以先用着，面板不是必须的；**

```yaml
页面清单:
  - 登录页: 管理员登录 + 双因素认证（可选）
  - 仪表盘: 今日调用量 / Token 消耗 / 活跃 Key / 告警概览
  - 工作流管理: 工作流列表 / 启用/禁用 / 执行记录 / 错误概览
  - Key 管理: Key 创建 / 吊销 / 配额设置 / 权限配置
  - 审计日志: 按时间 / Key / 模型 / 状态的日志查询
  - 设置页: 系统配置 / 邮件服务 / 告警通道

工时分布:
  Day 1-3: 项目初始化 + 登录 + 路由 + 布局
  Day 4-6: 仪表盘 + 数据可视化图表
  Day 7-9: 工作流管理
  Day 10-12: Key 管理 + 审计日志
  Day 13-14: 设置 + 告警配置
  Day 15: 集成测试 + 部署配置 + 文档
```

### 数据库设计（2 天）

**前提假设是你在 Form-B 已有的 PG 表结构上做优化，而非从零设计：**

```yaml
工时分布:
  Day 1: 分区表设计 + pgvector 索引 + 归档策略 SQL
  Day 2: 备份策略脚本（WAL 归档 + pg_dump + 自动清理）+ 压测

核心产出:
  - execution_entity 表的分区创建脚本
  - embedding_cache 表的向量索引创建
  - 数据清理 cron job 脚本
  - 备份与恢复 SOP 脚本
```

## 影响工时的关键因素

```yaml
factor_1: 团队背景
  - 有 Go 生产经验: 估算 * 0.8
  - 首次接触 Go: 估算 * 1.5
  - 团队有 K8s 经验: K8s 部署部分 * 0.6
  - 首次部署 K8s: K8s 部署部分 * 2.0

factor_2: AI 使用熟练度
  - 能有效编写提示词、迭代解决问题: 估算 * 0.7
  - 第一次用 AI 辅助编码: 估算 * 1.3
  - 会用 Cursor/GitHub Copilot 深度集成: 估算 * 0.6

factor_3: 产品化程度
  - 内部使用/小团队: 估算 * 1.0
  - 给客户部署（需要文档 + 运维手册）: 估算 * 1.3
  - 商业产品（需要完整测试 + 安全审计 + 国际化）: 估算 * 2.0

factor_4: 开发者人数影响
  - 1 人独立开发: 总日历天数 ≈ 总人天（串行）
  - 2 人并行: 总日历天数 ≈ 总人天 * 0.65
  - 3 人并行: 总日历天数 ≈ 总人天 * 0.5
  - 注意: Python 版两人并行最快（~5 个组件，2 人 x 3 周）
```

## 典型场景估算

### 场景 1：单人全栈（创始人/独立开发者）

```yaml
角色: 懂一些 Go/Python，会用 AI 辅助编码
目标: 替换 n8n + 自研 auth-server + ai-gateway
时间线: 6-8 周

Week 1: 技术选型 + 环境搭建
Week 2-3: auth-server + ai-gateway
Week 4-5: n8n 替代引擎（最小功能集）
Week 6: 集成测试 + bug 修复
Week 7-8: Dockerfile + 部署 + 文档

推荐策略:
  1. 不要先做 n8n 替代
  2. 先用 Form-B 跑 n8n，同时重写 auth-server + ai-gateway
  3. 等 auth 和 gateway 稳定了再替换 n8n
  4. 可以在生产环境中混搭（n8n + 自研服务一起跑）
```

### 场景 2：三人团队（技术负责人 + 后端 + 前端）

```yaml
角色:
  - 技术负责人: 架构 + 核心模块 + AI 驱动
  - 后端: auth-server(Golang) + ai-gateway(Python)
  - 前端: 管理面板(React)
时间线: 3-4 周

Week 1:
  技术负责人: 部署 Form-B + 技术报告 + 范围切分
  后端: auth-server 开发
  前端: 面板项目初始化 + 登录页 + 仪表盘

Week 2:
  技术负责人: n8n 替代引擎框架 + AI 辅助代码生成
  后端: ai-gateway 开发 + 集成 auth-server
  前端: 工作流管理页 + Key 管理页

Week 3:
  技术负责人: n8n 替代引擎完善 + 集成测试
  后端: 压力测试 + 安全测试 + bug 修复
  前端: 审计日志页 + 设置页 + 集成

Week 4:
  全局: 端到端测试 + 性能调优 + 部署 + 文档

推荐策略:
  这是一个合理的团队配置，3-4 周可以交付生产级系统
  注意让技术负责人不要陷入编码细节——他应该用 AI 加速
```

### 场景 3：企业项目（5 人 + 全功能）

```yaml
角色: 5 人团队（架构师 + 2 后端 + 1 前端 + 1 QA/Ops）
目标: 全功能自研 + K8s 部署 + 商业级质量
时间线: 6-8 周

推荐策略:
  - 不要同时做 K8s 迁移和代码重写——这是最常见的失败模式
  - 先完成代码重写（docker-compose 部署），
  - 稳定运行 1 个月后再迁移到 K8s
  - 如果必须做 K8s，做好 2 周的额外缓冲期
```
